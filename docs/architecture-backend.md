## 2. API Layer

### Application Bootstrap (`main.py`)

The FastAPI app initializes in a `lifespan` context manager:

- Creates `data/uploads/`, `data/stems/`, `data/remixes/` directories
- Creates two `ThreadPoolExecutor` instances:
  - `app.state.executor` — max 1 worker; serializes remix processing (only one remix at a time)
  - `app.state.sse_executor` — max 4 workers; handles blocking SSE queue reads
- Creates `app.state.sessions: dict` — in-memory session registry (no DB)
- Creates `app.state.sessions_lock` — threading lock for session dict access
- Creates `app.state.processing_lock` — threading lock enforcing single-remix-at-a-time

CORS is configured to allow `http://localhost:5173` (the React frontend dev server) by default.

Static files (the fallback HTML UI) are mounted at `/` **after** all API routers, so the catch-all doesn't swallow API routes.

### All Endpoints

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/health` | Returns `{"status": "ok"}` |
| `POST` | `/api/remix` | Upload two songs (multipart), start async pipeline, return `{"session_id": "<uuid>"}` |
| `POST` | `/api/remix/youtube` | Accept two YouTube URLs + prompt (JSON), download and pipeline |
| `GET` | `/api/remix/{id}/progress` | SSE stream of pipeline progress events |
| `GET` | `/api/remix/{id}/status` | JSON snapshot: `{status, remix_path, explanation, last_event}` |
| `GET` | `/api/remix/{id}/audio` | Serve the rendered `remix.mp3` (FileResponse) |
| `GET` | `/api/remix/{id}/stems` | List available stem WAV files for both songs |
| `GET` | `/api/remix/{id}/stems/{song}/{stem}` | Serve a raw WAV stem file |

### Request/Response Shapes

**POST /api/remix** (multipart form):
- `song_a`: UploadFile (`.mp3` or `.wav`, max 50MB)
- `song_b`: UploadFile (`.mp3` or `.wav`, max 50MB)
- `prompt`: str (optional, default `""`)
- Response: `{"session_id": "uuid4"}`
- Error 409 if another remix is already in progress
- Error 413 if file exceeds size limit
- Error 422 if file extension not allowed

**POST /api/remix/youtube** (JSON body):
```json
{"url_a": "https://youtube.com/...", "url_b": "https://youtube.com/...", "prompt": "..."}
```
- Response: `{"session_id": "uuid4"}`
- Validates YouTube URLs before acquiring processing lock

**GET /api/remix/{id}/progress** (SSE):
```
data: {"step": "separating", "detail": "Extracting stems...", "progress": 0.10}
data: {"step": "keepalive", "detail": "", "progress": -1}
data: {"step": "complete", "detail": "Remix ready!", "progress": 1.0, "explanation": "...", "warnings": [], "usedFallback": false}
data: {"step": "error", "detail": "...", "progress": 0}
```
Step values: `downloading` | `separating` | `analyzing` | `interpreting` | `processing` | `rendering` | `complete` | `error` | `keepalive`

### Middleware and Auth

- **CORS**: `fastapi.middleware.cors.CORSMiddleware` — `allow_origins=settings.cors_origins`
- **No authentication** — single-user proof of concept
- **Session ID validation**: UUIDs validated with `uuid.UUID()` before path traversal
- **Path traversal protection**: All file paths resolved with `.resolve()` and checked against `data_dir`

## 3. Audio Processing Pipeline

The main pipeline (`services/pipeline.py → run_pipeline()`) executes 17 steps in a background thread. Steps 1 and 2 overlap concurrently; all other steps are sequential.

### Step-by-Step Breakdown

| Step | Name | Description |
|------|------|-------------|
| 1 | Stem separation | Both songs separated concurrently in a thread pool (up to 6 workers) |
| 2 | Audio analysis | `analyze_audio_full()` on both originals simultaneously — overlapped with separation |
| 3 | BPM reconciliation | `reconcile_bpm()` — handles double/half BPM detection artifacts |
| 3.5 | Song structure | `analyze_stems()` per song: per-bar RMS, vocal activity, sections, vocal gaps |
| 3.7 | Lyrics mapping | Map lyric timestamps or plain lyrics to bar numbers |
| 4 | LLM interpretation | `interpret_prompt()` — Anthropic `claude-sonnet-4-20250514` via `tool_use` → `RemixPlan` |
| 4.5 | Taste stage | Optional: generate 8-12 candidate plans, filter, score, select best (gated by `ab_taste_model_v1` flag) |
| 5 | Source assignment | Fixed: Song A = vocals, Song B = instrumentals |
| 6 | Stem loading | `validate_stem()` — load WAVs, resample to 44.1kHz, ensure stereo float32 |
| 7 | Trimming | Slice stems to plan's `start_time_*` / `end_time_*` ranges |
| 7.5 | Silent stem filter | Exclude stems below -50 LUFS (near-silent after separation) |
| 7.7 | Vocal bandpass | 150Hz–16kHz zero-phase Butterworth on vocal stem |
| 7.75 | Corrective EQ | Per-stem EQ presets via Pedalboard (gentle cuts/boosts) |
| 8 | Tempo plan | `compute_tempo_plan()` — target BPM, which sides to stretch, stretch limits |
| 9 | Tempo stretch | `rubberband_process()` — all stems in parallel (6-worker pool), CLI subprocess |
| 10 | Beat re-detection | Re-detect beats on stretched instrumental sum; fallback to scaled original grid |
| 11 | Vocal compression | 3:1 RMS compressor, -20dB threshold, 3dB makeup, 10ms attack, 80ms release |
| 11.5 | Level matching | `cross_song_level_match()` — LUFS equalization +2dB offset toward instrumental |
| 11.8 | Drum/bass pre-limit | True-peak limiter: drums -6dBTP, bass -4dBTP |
| 12 | Arrangement render | `render_arrangement()` → vocal bus + instrumental bus |
| 12.5 | Spectral ducking | `spectral_duck()` — -3.5dB reduction in 300–3kHz when vocals active |
| 13 | Bus sum | `vocal_bus + ducked_instrumental` |
| 13.7 | Auto-leveler | `auto_level()` — 4s window, max ±2.5dB/1.5dB, detector=instrumental bus |
| 14 | Mastering | `master_static()`: constrained LUFS normalize → true-peak limiter (-1.0dBTP) |
| 14.5 | LUFS correction | Post-mastering correction loop: measure → boost up to 3dB → second limiter pass |
| 14.6 | Safety soft clip | tanh soft clip at -1.0dBTP ceiling, 2dB knee |
| 15 | Fades | 2s fade-in (cosine²), 3s fade-out |
| 16 | MP3 export | ffmpeg subprocess: float32 WAV → libmp3lame 320kbps, no dither |

### Key Processing Invariants

- **All audio is float32** throughout. Never int16 mid-pipeline.
- **44.1kHz / stereo** after `validate_stem()`. Analysis runs at 22050Hz internally.
- **No pydub** — pydub quantizes to 16-bit. All mixing is numpy; export is ffmpeg subprocess.
- **rubberband time ratio**: `source_bpm / target_bpm` (NOT the inverse)
- **Spectral ducking** uses a **copy** of instrumental bus — the original un-ducked bus is preserved for the auto-leveler detector signal

## 4. AI / LLM Integration

### Model and Config

- **Model:** `claude-sonnet-4-20250514` (configurable via `llm_model` in `.env`)
- **API key:** `ANTHROPIC_API_KEY` env var
- **Timeout:** 20 seconds per request
- **Retries:** 1 retry on failure
- **Client:** `anthropic.Anthropic()` (synchronous SDK)

### Tool Use Schema

The LLM is constrained to produce output by calling a single tool `create_remix_plan` with a JSON schema defining:

```
RemixPlan fields:
- start_time_vocal, end_time_vocal (float, seconds)
- start_time_instrumental, end_time_instrumental (float, seconds)
- sections[] (2-20 items):
    - label: "intro"|"verse"|"breakdown"|"drop"|"outro"
    - start_beat, end_beat (integer, multiple of 4)
    - stem_gains: {vocals, drums, bass, guitar, piano, other} — all 0.0-1.0
    - transition_in: "fade"|"crossfade"|"cut"
    - transition_beats: 0-8
- tempo_source: "song_a"|"song_b"|"average"
- key_source: "song_a"|"song_b"|"none"
- explanation: string (≤500 chars, user-facing)
- warnings: string[]
```

### System Prompt Architecture (Prompt Caching)

Two content blocks for Anthropic prompt caching:

**Block 1 (static, cached):** Seven static sections covering role constraints, failure mode guards, mixing advisory rules, section rules, genre guidance, stem artifact awareness, and explanation format.

**Block 2 (dynamic, uncached):** Six per-request sections including tempo/stretch advisory, arrangement templates with beat budget, song data (5 layers), cross-song analysis, and lyrics.

**5-Layer Song Data sent to LLM:**
1. Song overview: BPM, key, duration, beat count, energy profile
2. Song structure: Detected sections with start/end bars, labels, energy levels, vocal status, annotations
3. Stem character: Per-stem energy levels and vocal prominence
4. Cross-song relationships: Loudness diff, stretch %, key compatibility
5. Lyrics: Bar-mapped lyrics for arrangement-aware decisions

### Fallback Behavior

On any LLM failure (timeout, API error, schema validation failure, arrangement too short), falls back to `generate_fallback_plan()` — a deterministic algorithm that produces a 5-section arrangement (intro → verse → breakdown → drop → outro). `plan.used_fallback = True` when this path is taken.

### Taste Training System (Optional, `ab_taste_model_v1` flag)

When enabled:
1. **Candidate generation**: 8-16 `RemixPlan` variants across 4 arrangement families (Standard Arc, Hook-First, DJ Lift, Quick Hit)
2. **Constraint filtering**: 14 hard constraints per candidate
3. **Scoring**: 7-dimension heuristic rubric with CatBoost ML model stubbed for Phase 2+
4. **Feature extraction**: 29 Tier 1 features, versioned by manifest hash
5. **Circuit breaker**: 5 consecutive failures → 10-minute cooldown
6. **Timeout**: 400ms or falls back

## 5. Data Flow: Request Lifecycle

### File Upload Path

```
Client → POST /api/remix (multipart)
  → Validate file extensions (.mp3, .wav)
  → Acquire processing_lock (non-blocking; 409 if busy)
  → Save files to data/uploads/{session_id}/
  → Create SessionState (status="queued")
  → Submit _pipeline_wrapper to executor (background thread)
  → Return {"session_id": "..."}
```

### YouTube Path

```
Client → POST /api/remix/youtube (JSON)
  → Validate YouTube URLs (SSRF prevention)
  → Acquire processing_lock
  → Create session
  → Parallel download Song A + Song B via yt-dlp
  → Run same pipeline as file upload path
```

### SSE Event Delivery

```
Client → GET /api/remix/{id}/progress
  → StreamingResponse(media_type="text/event-stream")
  → Loop: queue.get(timeout=5), emit keepalive on timeout
  → Break on "complete"/"error"
  → 20-minute safety cap
```

## 7. Storage

**No database.** All state is file-based in `data/` and in-memory `app.state.sessions`.

### Directory Layout

```
data/
├── uploads/{session_id}/       # Raw uploaded/downloaded audio
├── stems/{session_id}/         # Separated stems (~40MB per stem, float32 WAV)
│   ├── song_a/
│   └── song_b/
├── remixes/{session_id}/       # Final output (~5-15MB at 320kbps)
│   └── remix.mp3
└── stem_cache/{sha256_hex}/    # Reusable cached stems, LRU-evicted
```

### In-Memory Session State

```python
@dataclass
class SessionState:
    status: str              # "queued" | "processing" | "complete" | "error"
    events: queue.Queue      # maxsize=100; SSE event FIFO
    created_at_mono: float   # time.monotonic() for age tracking
    remix_path: str | None   # Absolute path to remix.mp3 when done
    explanation: str | None  # LLM explanation shown to user
    last_event: dict | None  # Most recent event (reconnection recovery)
```

Sessions disappear on server restart. The `remix.mp3` file survives independently.

### Stem Cache

- **Key:** SHA-256 hex digest of input audio file
- **Write:** Atomic via temp dir + `os.rename()`
- **Eviction:** LRU by directory mtime; oldest entries deleted until total ≤ `stem_cache_max_gb`
- **Thread safety:** Concurrent writers produce identical content; last rename wins

## 8. SSE / Real-Time Progress

### Architecture

```
Pipeline thread (sync)          SSE generator (async)
    ↓                               ↓
emit_progress(queue, event) →→→ session.events.Queue
                                    ↓
                              run_in_executor(sse_executor, queue.get(timeout=5))
                                    ↓
                              yield f"data: {json.dumps(event)}\n\n"
```

### Key Design Decisions

- **Non-blocking emit:** `queue.put_nowait()` — drops non-terminal events if queue full. Terminal events drain one old event first.
- **Keepalive:** Every 5 seconds of silence, emit `{"step": "keepalive"}` to prevent proxy timeouts.
- **Reconnection:** `session.last_event` stores the most recent event for immediate send on reconnect.
- **20-minute safety cap** on stream duration.

## 9. Key Abstractions

### `RemixPlan` (models.py)
The central data structure. Produced by the LLM (or fallback), consumed by the renderer and DSP chain. Contains source time windows, ordered sections with per-stem gains and transitions, tempo/key matching strategy, explanation, and warnings.

### `AudioMetadata` (models.py)
Per-song analysis: BPM, beat frames, key, scale, confidence, duration, total beats, stem analysis, song structure.

### Separation Dispatcher (`separation.py`)
Single function `separate_stems(audio_path, output_dir)`: check cache → dispatch to modal or local → write to cache. Downstream code never knows which backend was used.

### `beats_to_samples()` (renderer.py)
Converts beat indices to sample positions using the actual detected beat grid (not constant-BPM math). Extrapolates beyond last beat using average inter-beat interval.

### `_build_gain_curves()` (renderer.py)
Pre-computes continuous per-stem gain arrays using cosine interpolation at section boundaries. Eliminates "fade from silence" artifacts.

## 11. Key Conventions and Gotchas

1. **Song A = vocals, Song B = instrumentals** — Fixed convention via `VOCAL_SOURCE = "song_a"` constant.
2. **`data_dir` is relative** — resolves from the server's working directory.
3. **StaticFiles must be mounted last** — otherwise it swallows `/api/*` routes.
4. **Processing lock is non-blocking** — 409 immediately if busy (one remix at a time).
5. **rubberband time ratio is `source_bpm / target_bpm`** — NOT the inverse.
6. **auto_level detector signal** — must use `instrumental_bus` (not summed mix) to avoid volume-dip regression.
7. **LUFS measurement** — `pyloudnorm` expects stereo `(N, 2)` shape.
8. **Pedalboard channel convention** — expects `(channels, samples)`, pipeline uses `(samples, channels)`.
9. **Stem cache concurrent writes** — safe via atomic `os.rename()`.
