## 3. State Management

### Architecture: Single Reducer with Discriminated Union Phases

No Context, Redux, Zustand, or other global state library. State lives entirely in `RemixSession` via `useReducer(remixReducer, initialState)`, with `dispatch` threaded down as props.

### AppState — Phase Machine

| Phase | Description | Key fields |
|---|---|---|
| `idle` | Form ready for input | `songA`, `songB`, `prompt` |
| `uploading` | File upload in flight | same + `uploadProgress: number` |
| `submitting` | YouTube submit in flight | `songA`, `songB`, `prompt` |
| `processing` | SSE stream active | `sessionId`, `progress: ProgressEvent`, form fields |
| `ready` | Remix complete, playable | `sessionId`, `explanation`, `warnings`, `usedFallback` |
| `error` | Error state | `message`, preserved form fields |

### Reducer Actions (14 total)

```
SET_SONG_A, SET_SONG_B                    — file-based song input
SET_YOUTUBE_URL_A, SET_YOUTUBE_URL_B      — YouTube-based song input
CLEAR_SONG_A, CLEAR_SONG_B               — remove a song slot
SET_PROMPT                               — text prompt update
START_UPLOAD                             — idle → uploading
START_SUBMIT                             — idle → submitting
UPLOAD_PROGRESS                          — update upload %
UPLOAD_SUCCESS / SUBMIT_SUCCESS           — → processing
PROGRESS_EVENT                           — SSE event arrived
REMIX_READY                              — → ready
ERROR / RETRY / CANCEL / RESET           — error/recovery/navigation
```

Each action validates the current phase before transitioning. `ERROR` preserves form data. `RETRY` → `idle` (keeps form data). `RESET` → full initial state.

### Local State (Component-Scoped)

| Component | Local state | Purpose |
|---|---|---|
| `RecordPlayerView` | `phase: AnimationPhase` | `placing → idle → playing → paused` |
| `SongUpload` | `mode`, `urlInput`, `dragOver`, `error` | File vs YouTube toggle; validation |

| `RemixPlayer` | `confirmNew` | Two-step confirmation for new remix |

## 4. API Integration

All API calls centralized in `src/api/client.ts`. Components never contain raw `fetch` calls.

**Base URL:** `/api` — proxied to `http://localhost:8000` by Vite in dev.

### Endpoints

| Function | Method | Path | Notes |
|---|---|---|---|
| `createRemix()` | POST | `/api/remix` | Uses `XMLHttpRequest` for upload progress; multipart form |
| `submitYouTubeRemix()` | POST | `/api/remix/youtube` | Uses `fetch`; JSON body |
| `connectProgress()` | GET | `/api/remix/{id}/progress` | Returns native `EventSource` |
| `getAudioUrl()` | — | `/api/remix/{id}/audio` | Returns URL string for `<audio src>` |
| `getSessionStatus()` | GET | `/api/remix/{id}/status` | For reconnection (not currently used in components) |

### External API

`SongUpload` calls `https://noembed.com/embed?url=...` directly for YouTube title and thumbnail (public oEmbed proxy, no auth).

### Error Handling

`RemixSession.formatError` maps `CreateRemixError` to user-facing strings:
- HTTP 429: "Another remix is being created."
- HTTP 413: "File too large. Maximum 50MB per song."
- HTTP 422: uses `body.detail`
- Network/timeout: generic messages

## 5. Audio Playback

### Approach: HTML5 `<audio>` Element

Standard HTML5 `<audio>` — no Web Audio API, no AudioContext, no analyzers.

### `useAudioPlayer` Hook

**State tracked:** `isPlaying`, `isPaused`, `isLoaded`, `isBuffering`, `currentTime`, `duration`, `progress (0-1)`, `error`

**Controls:** `play()`, `pause()`, `seek(time)`, `rewind(seconds)`, `audioRef`

**Events wired:** `loadedmetadata`, `timeupdate`, `play`, `pause`, `ended`, `waiting`, `playing`, `error`

**Auto-play:** If `autoPlay: true`, auto-plays once `isLoaded` becomes true.

### `RecordPlayerView` — Playback Coordination

Bridges audio state and visual animation via `AnimationPhase` state machine:

```
placing (1500ms fade-in) → idle → playing ↔ paused
                                         ↓ (track ends)
                                         idle
```

**Tonearm angle:** Parked at 0°; playing sweeps 25°–32° as track progresses.

**Play flow:** Phase → `playing` → 400ms delay → `play()` on audio → if browser blocks, revert phase.

**Track end:** Detected via `currentTime >= duration - 0.1`.

## 6. UI/UX Flow

### Phase-by-Phase Rendering

| Phase | Renders | Key UX |
|---|---|---|
| `idle` | `RemixForm` | Two song slots (file or YouTube) + prompt + "Create Remix" button |
| `uploading` | `RemixForm` (disabled) | Upload progress bar |
| `submitting` | `RemixForm` (disabled) | No progress bar |
| `processing` | `ProgressDisplay` | VinylMergeAnimation + step label + cancel link |
| `ready` | `RemixPlayer` | Turntable + explanation + warnings + "Create New Remix" |
| `error` | Inline error panel | Red-bordered box + "Try Again" button |

### Song Input UX

**File mode (default):** Drag-and-drop zone, click-to-browse, accepts `.mp3`/`.wav` up to 50MB.

**YouTube mode:** URL text input, validates against YouTube regex, fetches oEmbed preview with thumbnail and title.

**Mixed input restriction:** File + YouTube combo is blocked; all-files or all-YouTube only.

### VinylMergeAnimation (Processing)

3-layer SVG animation driven by `progress: number` (0–1):

| Range | Layer 1 (vinyl records) | Layer 2 (gooey blob) | Layer 3 (marble disc) |
|---|---|---|---|
| 0–20% | Visible, moving together | Hidden | Hidden |
| 20–40% | Overlapping | Hidden | Fading in |
| 40–55% | Fading out | Fading in | Growing |
| 55–70% | Gone | Fading out | Bright |
| 70–100% | Gone | Gone | Full; grooves + glow at 95% |

SVG filter attributes updated imperatively via `setAttribute()` on refs.

## 9. Key Abstractions

### Custom Hooks

#### `useRemixReducer`
Pure state transition logic. Exports `remixReducer` and `initialState`. No side effects.

#### `useRemixProgress(sessionId, dispatch)`
Opens `EventSource` when `sessionId` is non-null. Per-step timeouts (120s download, 180s separation, 60s analysis/interpret, 120s processing). Reconnects up to 4 times. Enforces monotonic progress. Delays `REMIX_READY` by 800ms for animation completion.

#### `useFormPersistence(state, dispatch)`
**IndexedDB** (`musicSpinner` db): persists `File` objects as `{ blob, name, type }`. **sessionStorage**: persists prompt and YouTube URLs. Clears all storage when phase reaches `ready`.

#### `useAudioPlayer(options)`
Wraps HTML5 `<audio>` element. Returns flat object with state + controls. All controls are `useCallback`-stable.

#### `useOEmbed(url)` (inline in SongUpload)
Not exported. Fetches YouTube metadata from noembed.com. Uses cleanup pattern to prevent stale updates.

## 12. Key Design Decisions

1. **No Context.** Tree is shallow (max 3 levels), so prop drilling is fine.
2. **Phase machine over ad-hoc booleans.** Discriminated union makes illegal states unrepresentable.
3. **XHR over fetch for upload.** `fetch` doesn't expose upload progress natively.
4. **Imperative SVG filter updates.** CSS transitions can't animate SVG filter primitive attributes.
5. **Monotonic progress enforcement.** `maxProgressRef` prevents backward jumps (important for YouTube→pipeline handoff).
6. **IndexedDB for file persistence.** `File` objects can't be serialized to localStorage/sessionStorage.
7. **Touch vs. hover controls.** `FloatingControls` detects touch devices dynamically.
8. **Animation-first completion.** 800ms delay before `REMIX_READY` lets the merge animation reach its final state.
